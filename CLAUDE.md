# digikam-web

A web backend (Rust) that serves the photos in a [Digikam](https://www.digikam.org/) SQLite database over an HTTP/JSON API. **Read-only by default**: the Digikam DB is only writable when started with the opt-in `--allow-writes`, which enables the write endpoints (rating changes, tag editing, and moving photos between albums ‚Äî the move also renames the file on disk; uploads are planned).

## Build & run

Everything goes through Nix (flakes). nixpkgs is pinned to **26.05**; the build uses [crane](https://github.com/ipetkov/crane) + [rust-overlay](https://github.com/oxalica/rust-overlay).

```bash
nix run                              # build + run with defaults
nix develop --command cargo run      # iterate inside the dev shell
nix develop --command cargo test     # unit + API integration tests (seeded temp SQLite)
nix build                            # produce ./result/bin/digikam-web
nix flake check                      # build + clippy (-D warnings)
```

Config (CLI flags or env vars):
- `--database` / `DIGIKAM_DB` ‚Äî path to `digikam4.db` (default `~/.local/share/digikam/db/digikam4.db`).
- `--listen` / `LISTEN_ADDR` ‚Äî bind address (default `127.0.0.1:8080`). Plain HTTP/1.1 (`axum::serve`); terminate TLS in a reverse proxy (e.g. nginx) in front of it.
- `--thumbnail-database` / `THUMBNAIL_DB` ‚Äî path to `thumbnails-digikam.db` (default: alongside `--database`). Optional; if missing, `/thumbnail` returns `404`.
- `--trace-sql` / `TRACE_SQL` ‚Äî log every executed SQL statement (with bound values inlined) at `info` under the `digikam_web::sql` target. Off by default.
- `--web-database` / `WEB_DB` ‚Äî path to the **writable** bookmarks DB (`web.sql`, default: alongside `--database`; created if missing). Without `--allow-writes`, this is the *only* DB we write to.
- `--allow-writes` / `ALLOW_WRITES` ‚Äî opt in to **writing the Digikam DB** (the main pool opens read-write instead of read-only + `query_only`). Off by default; without it the write endpoints (`PATCH /api/photos/:id`) return `403`.

> Nix flakes only see git-tracked files. After adding/renaming a file, `git add` it before `nix build`/`nix develop`, or Nix won't find it (and crane needs `Cargo.lock` tracked).

## API

All endpoints are served under the `/api` prefix, except the root-level `/random` (last row).

| Route | Notes |
|-------|-------|
| `GET /api/photos?album=&tags=&recursive=&min_rating=&images=&video=&aspect=&sort=&limit=&offset=` | Filtered, paginated list. `Page<PhotoSummary>` = `{incomplete, limit, offset, items}` (`incomplete` = more rows exist beyond this page ‚Äî one extra row is fetched to detect it). `PhotoSummary.is_video` is true for videos (Digikam `category=2`). Each `PhotoSummary` carries both `modification_date` and `creation_date` (so the client can day-group by whichever the sort uses). `images`/`video` are the media-type filter (both default `true`; `=false` excludes that type). `aspect` is `all` (default) / `portrait` (`height>=width`) / `landscape` (`width>=height`); squares match both. `sort` is `modified` (default) / `created` / `name` (see Query semantics ‚Üí Ordering). An empty/absent `album` (non-recursive) returns `items: []`. |
| `GET /api/photos/:id` | `PhotoDetail` = the `PhotoSummary` fields (which now include `creation_date`) plus `file_path`, `description`, `latitude`/`longitude`, and `tags`. `file_path` is the **absolute path of the original on the server** (album-root base + `relativePath` + name; `null` if the root is unknown) ‚Äî the info panel's copy-path button copies it. `creation_date` (a `PhotoSummary` field) is `ImageInformation.creationDate` (Digikam's import/EXIF time ‚Äî distinct from the `modificationDate` the app groups by under the default sort); `description` is the image's `ImageComments` (caption/title/headline/imported EXIF-JFIF comments) concatenated with newlines (ordered by `type, language, id` via a `group_concat` subselect in the one query; `null` when none); `latitude`/`longitude` are `ImagePositions.latitudeNumber`/`longitudeNumber` (null when absent). `tags` are **absolute paths** (`/vacation/2020/beach`, built by walking the `pid` chain), sorted `COLLATE NOCASE`, with Digikam's internal tags (the `_Digikam_Internal_Tags_` subtree ‚Äî Color/Pick labels, version history) excluded. The lightbox info panel fetches this lazily (only while open) and caches it per id. |
| `PATCH /api/photos/:id` | **Write endpoint** (needs `--allow-writes`, else `403`): partial update of a photo's Digikam metadata, applied in **one transaction** (`unchecked_transaction` ‚Äî all or nothing). Scalar fields are tri-state ‚Äî **absent** = leave unchanged, `null` = clear, value = set (`Option<Option<‚Ä¶>>` via the `double_option` deserializer in [src/models.rs](src/models.rs) `PatchPhoto`) ‚Äî so `{}` is a valid no-op (`204` without touching the DB). `rating`: `{"rating": 0..5}` sets, `{"rating": null}` clears to Digikam's "unrated" (`-1`); out-of-range ‚Üí `422` (the shared `Rating` type). The write is an **upsert** into `ImageInformation.rating` (an image may have no row yet). `tags_add` / `tags_remove`: arrays of **tag ids** (from `/api/tags`) to link/unlink in `ImageTags` ‚Äî plain collection deltas (absent/empty = no change), composable with `rating` in one request. Already-present adds and absent removes are silent no-ops (idempotent); an unknown or **Digikam-internal** tag id (the `_Digikam_Internal_Tags_` subtree) ‚Üí `422` (`AppError::Unprocessable`, checked via `checked_tag_path` which also resolves the path for logging). `album`: target album id (from `/api/albums`) ‚Äî **moves** the photo: `Images.album` **and the file on disk**. Same-filesystem `rename` only (atomic; content + mtime + `uniqueHash` untouched, so thumbnails keep working); the current album is a silent no-op; `422` for an unknown album id / unresolvable root; `409` for a name collision in the target, a missing target directory (DB/FS drift is refused, not papered over), a cross-device move, or a source file missing on disk (`rename_error`). Ordering: the `UPDATE` runs inside the transaction, then the `rename`, then the commit ‚Äî a rename failure rolls the whole patch back (incl. any rating/tag changes in the same request), a failed commit renames back (best effort). The move is processed **last** in the handler for exactly this reason. Unknown/trashed photo id ‚Üí `404`, success ‚Üí `204`. Every **actual** change is logged at `info` with the image id + absolute path: rating old‚Üínew, each tag add/remove with the tag's path **and numeric id**, and each move with the old + new absolute paths. Metadata writes touch the **DB only** ‚Äî not the file's XMP/EXIF (like Digikam with "write metadata to files" off); only a move touches disk; a concurrently-running Digikam won't auto-refresh externally-changed metadata. |
| `GET /api/photos/:id/file` | Original bytes, range-aware (via `tower_http::services::ServeFile`). Sends a strong `ETag` from the image's `uniqueHash`; a matching `If-None-Match` (or `*`) returns `304`. `Content-Disposition: inline` carries the original filename (+ RFC 5987 `filename*`) so saving from the browser keeps the real name. |
| `GET /api/photos/:id/reverse-search?engine=yandex` | Reverse-image-search the original on Yandex; **302-redirects** to the results page. The **server** does the upload (not the browser): the image URLs may not be world-readable, so a `?url=` search Yandex would have to fetch back is unreliable (must send the **bytes**), and Yandex sends no CORS headers + returns only a transitional "candidate" page, so the browser can't read the CBIR id. The handler POSTs the file as multipart `upfile` to Yandex's JSON endpoint (`reqwest`, rustls), reads `blocks[0].params.cbirId`, and redirects to `‚Ä¶/search?rpt=imageview&cbir_id=‚Ä¶&cbir_page=similar`. The only outbound network call this app makes. `engine` defaults to `yandex` (only value supported ‚Üí else `400`); unknown id `404`; Yandex failure `502`. |
| `GET /api/photos/:id/thumbnail` | Digikam's stored thumbnail as-is: the **raw PGF blob** from `thumbnails-digikam.db` (looked up by `uniqueHash`+`fileSize`), for the client to decode in wasm (see [nix/webpgf.nix](nix/webpgf.nix)). Strong `ETag` (+ `If-None-Match`‚Üí`304`) and a 1-year `immutable` `Cache-Control`; `X-Orientation` header carries Digikam's `orientationHint` (EXIF orientation) for client-side rotation. `404` when the thumbnails DB is absent / the image has no cached thumbnail ‚Üí client falls back to `/file`. |
| `GET /api/albums` | Flat list of all albums (`{id, path, root}`). |
| `GET /api/subalbums?album=/Root/rel&min_rating=&images=&video=&aspect=&tags=&sort=` | Direct sub-albums of an album as `[{name, path, photo_count, cover: {id, name} \| null}]`, sorted by most recent photo (newest first) ‚Äî **except `sort=name`, which orders them alphabetically** (`COLLATE NOCASE`). An absent/empty `album` lists the album roots. The **cover follows the same `sort` as the grid**: the sub-album's whole-subtree item that sorts first ‚Äî newest by `modificationDate` (default), newest by `creationDate` (`sort=created`), or **alphabetically-first by name** (`sort=name`). It can be an image **or** video (videos have stored thumbnails the client renders); `Cover.is_video` flags a video cover (the client then omits its `data-full`). `photo_count` is the recursive count incl. videos. `min_rating` (0..=5), the `images`/`video` media-type filter, the `aspect` filter, and `tags` (same hierarchical semantics as `/photos`) constrain the cover, count, and which sub-albums appear alike. One query; albums with no matching photos anywhere are omitted. |
| `GET /api/tags` | Tag **tree** (`{id, name, children}`), internal tags excluded. |
| `GET /api/bookmarks` | Saved bookmarks `[{name, album, recursive, min_rating, include_images, include_video, aspect, tags, sort}]`, sorted by name (`COLLATE NOCASE`). `[]` if the bookmarks DB is unavailable. Each bookmark is a named album + filter snapshot (the filter fields are the flattened [`Filters`](src/models.rs), which includes `recursive` and `sort`). |
| `POST /api/bookmarks` | Create a bookmark from the same JSON shape (+ optional `overwrite`). Empty/over-long name ‚Üí `400`; bad `min_rating`/`aspect`/`sort` ‚Üí `422` (typed body validation); duplicate name without `overwrite` ‚Üí `409`; `overwrite:true` does `INSERT OR REPLACE`. |
| `DELETE /api/bookmarks/:name` | Remove a bookmark (idempotent ‚Üí `204`). |
| `GET /api/health` | Liveness. |
| `GET /random?album=&tags=&recursive=&min_rating=&images=&video=&aspect=` | **Root-level** (not under `/api`). (Accepts the same params as `/api/photos`, but `sort` is moot ‚Äî it's `ORDER BY random()`.) **307-redirect** to a random matching item's `/api/photos/:id/file`, with `Cache-Control: no-store` so every hit re-randomizes ‚Äî for screensavers / photo frames that just fetch a URL (pass `video=false` for an images-only frame, since the target `/file` of a video is the video bytes). Same album + filter params and semantics as `/api/photos` (incl. the empty-album-non-recursive rule); `404` when nothing matches. `random_photo_id` in [src/query.rs](src/query.rs) reuses `build_filter` with `ORDER BY random() LIMIT 1`. |

### Query semantics
- **`album=/Root/rel`** ‚Äî the first path segment is the `AlbumRoots.label`; the remainder is a `relativePath`. By default it matches **only that album** (photos directly in it). `/Photos` alone means the root album (`relativePath = "/"`). An **absent/empty `album`** (the virtual root) has no photos of its own, so `/api/photos` returns an **empty** list ‚Äî unless `recursive=true` (below).
- **`recursive`** ‚Äî a boolean: `?recursive=true` also includes all sub-albums; `?recursive=false` or absence keeps the default non-recursive behavior. With `?recursive=true`, `/Photos` selects the whole collection ‚Äî and an empty `album` with `recursive=true` selects **all** photos (every root).
- **`tags=a,b`** ‚Äî comma-separated tokens, **AND'd** (a photo must match every token; an unmatched token yields an empty result). Each token matches a tag **and all its subtags** ‚Äî **no substring match** (`foo` ‚âÝ `foobar`). A token starting with `/` is an **absolute path** (`/animals/cats` ‚áí that node + its subtree), matched **case-sensitively**, tag-only. Any other token is a **name**, matched **case-insensitively**, and matches either (OR): **(a)** any tag with that name at any level (+ subtree), **or (b)** photos in an **album named that token (or a sub-album thereof)** ‚Äî any `/`-delimited segment of the album's `relativePath` equals the token (so `vacation` ‚áí the `/places/vacation` tag tree **and** a `‚Ä¶/Vacation/‚Ä¶` album tree). Resolution = `resolve_tag_filter` in [src/query.rs](src/query.rs) (recursive `pid`-path CTE for `/`-tokens, `tag_ids_subquery` + `TagsTree` for the descendant closure, an album-segment `LIKE` for name tokens); per token it emits one `EXISTS`/`OR` predicate plus its **bound `?` parameters** (the token value + LIKE pattern), returned alongside the SQL like `build_filter` and spliced into the single main query (no per-token round-trips). Also a saved [`Filters`] field, so it applies to `/subalbums` and persists in bookmarks.
- **`min_rating=N`** ‚Äî minimum rating, `0..=5` (else `400`). Unrated images (Digikam stores `-1`) count as `0`, so `min_rating=0` includes everything and `min_rating>=1` excludes the unrated. Implemented as `max(ifnull(ii.rating,0),0) >= N`.
- **`images=` / `video=`** ‚Äî the media-type filter, two independent booleans, **both default `true`** (`=false` excludes that type; stock bool parsing, so the value is `true`/`false`). `video=false` ‚Üí only images (`category != 2`); `images=false` ‚Üí only videos (`category = 2`); both false ‚Üí empty. Like `min_rating`, it constrains the photo grid and the sub-album count/cover/visibility alike.
- **`aspect=`** ‚Äî aspect-ratio filter, an enum (so it can grow to exact ratios like `16:9`): `all` (default, no constraint) / `portrait` (`ii.height >= ii.width`) / `landscape` (`ii.width >= ii.height`); a **square** matches both (inclusive `>=`), an invalid value is a `400`. Items with NULL dimensions are excluded from `portrait`/`landscape` (the comparison is NULL) but kept under `all`. Constrains the grid and the sub-album count/cover/visibility alike.
- **Ordering / dates (`sort`)** ‚Äî `sort=modified` (default), `created`, or `name` (enum `Sort` in [src/query.rs](src/query.rs), modeled like `Aspect`; unknown value ‚Üí `400`, persisted in bookmarks). `modified` is newest-first by **`Images.modificationDate`** (`ORDER BY i.modificationDate DESC, i.id DESC`), `created` newest-first by **`ImageInformation.creationDate`**, `name` **ascending** by `Images.name` using a **natural ("version") sort** (`COLLATE NATSORT ASC, i.id ASC`) so numeric runs compare by value (`9_foo` < `10_bar`) ‚Äî a custom collation registered per-connection in `build_pool` ([src/db.rs](src/db.rs) `natural_cmp`), case-insensitive like `NOCASE` (named `NATSORT`, not `NATURAL`, which is a reserved SQL keyword). NULL dates sort last under the `DESC` orderings. The default deliberately uses the file modification date, **not** `creationDate` (Digikam's import time, rarely what you want for "newest"). The grid day-groups by the **sort's date column** (`creation_date` for `created`, else `modification_date`); **`name` sort renders a flat, ungrouped grid**. The sub-album **cover and recency follow the sort** too: the cover is each subtree's first item under the active sort (newest by the sort's date column, or alphabetically-first by name), and `sort=name` lists sub-albums alphabetically instead of by recency.
- **Paging** ‚Äî `limit` defaults to 25000, capped at 100000; `offset` defaults to 0.

## Frontend (HTML)

The `/photos` page is a **client-side SPA**. [src/web.rs](src/web.rs) serves a single **static shell** ‚Äî byte-identical for every URL ‚Äî rendered by `render` with no DB or album/filter logic: an empty navbar (`<span class="menu">` for the `‚ò∞` bookmarks menu, `<span class="crumb">` for the breadcrumb, `<span class="filter-menu">` for the funnel filters menu) plus empty `#subalbums` / `#photos` containers, the lightbox, and the inlined CSS/JS. The SPA ([src/web/](src/web/)) drives everything from in-memory **state** (`{album: segments[], minRating}`), initialized from the URL by `readUrl()` and updated on each navigation. It builds the navbar, sub-album tiles, and photo grid by fetching `/api/subalbums` + `/api/photos`, and **navigates in place**: clicking a breadcrumb / sub-album tile / `‚åÇ` / `‚òÖ` re-fetches and rebuilds the DOM (no page load) and updates the URL via `history.pushState`, so bookmarking + Back/Forward work.

| Route | Notes |
|-------|-------|
| `GET /photos`, `GET /photos/<album path>`, `GET /` | All serve the same static SPA shell. The client reads the album from the path (`/photos/Photos/Lego` ‚Üí `["Photos","Lego"]`; `/photos` = the virtual root) and filters from the query string, then renders the navbar + sub-album tiles + (for a real album) the day-grouped photo grid. See [The album page](#the-album-page). |

The shell is sent with `Cache-Control: private, max-age=3600` on success: cached an hour for navigations / back-forward, while a force-reload (Ctrl/Cmd+Shift+R) bypasses it. (Since the shell is now identical for every URL, this caches the one document across all albums.)

### The album page

Everything below is built **client-side** by the SPA ([src/web/](src/web/)) from `state`; in-page navigation re-renders without a page load. A single persistent runtime ‚Äî the thumbnail worker pool, the lightbox listeners, and the nav/popstate handlers ‚Äî is created once and reused across navigations; only the DOM is rebuilt (`render()` per navigation). A `renderToken` guard drops a fetch that resolves after a newer navigation.

- **Navbar (sticky** ‚Äî pinned to the top, the page scrolls underneath**)**: deliberately minimal (so it doesn't crowd on mobile) ‚Äî the `‚ò∞` bookmarks button (far left), a client-built breadcrumb starting with a `‚åÇ` home icon (‚Üí the root) then `‚Ä∫ Photos ‚Ä∫ Lego ‚Ä∫ Porsche911` (each segment a link to that ancestor album), and the funnel filters button (far right). **Alt+‚Üë** navigates to the parent album.
- **Two dropdown menus**, each `wireDropdown`-wired and built once into its static shell span (so they survive in-place re-renders). Their controls are the same state-reflecting `/photos` links as the rest of the UI, so `initNav` does the navigation and **clicks leave the menu open** (a panel ‚Äî tweak several filters, or jump between bookmarks, in a row). Each closes via its button, Esc, or an outside click (tested with `e.composedPath()`, not `contains(target)`, since a navigation re-render detaches the clicked control before the outside-click handler runs). Only one is effectively open at a time (opening/clicking one is an outside-click for the other).
  - **`‚ò∞` bookmarks menu** (far left, `.menu`, `initMenu`): a **Bookmarks** header with a `+` add (`+` prompts for a name, snapshots the current album + filters via `state`, `POST`s it; if the name exists it `confirm()`s and sends `overwrite`), then one row per bookmark ‚Äî a name link (built with `photosUrl`) + a `‚úï` delete (`confirm` ‚Üí `DELETE`). Sorted by name; `.menu-bookmarks` is (re)fetched from `/api/bookmarks` when opened / after a create/delete. (Bookmarks-only for now; other things may move in here later.)
  - **Funnel filters menu** (far right, `.filter-menu`, `initFilterMenu`): an inline-SVG funnel `.filter-btn` **highlighted gold when any filter is active** (`filtersActive()`, toggled by `renderMenuFilters` each render) ‚Äî at-a-glance feedback that filtering is in effect. Its `.menu-filters` panel is rebuilt every render from `state`: a **Filters** header carrying the `‚Ü∫` clear-all (links to the album with **every** filter reset; dimmed + inert when none active), then `Recursive` `[On|Off]` (extends the grid to all sub-albums' items, `?recursive=true`; sub-album tiles/counts are already recursive and `/api/subalbums` ignores it), `Stars` (five `‚òÖ`; star K ‚Üí `?min_rating=K`, clicking the active threshold clears it), `Media` (3-state segmented `üì∑üé•`/`üì∑`/`üé•` ‚áÑ `{includeImages, includeVideo}` ‚áÑ `images=/video=false`), `Aspect ratio` (3-state segmented `‚ñØ‚ñ≠`/`‚ñØ`/`‚ñ≠` ‚áÑ `state.aspect` ‚áÑ `aspect=portrait|landscape`), `Sort` (3-state segmented `Modified`/`Created`/`Name` ‚áÑ `state.sort` ‚áÑ `sort=created|name`), and `Tags` (a free-text comma-separated `<input>` ‚áÑ `state.tags` (array) ‚áÑ `tags=a,b`; commits on Enter/blur, navigating when changed). The active segment is an inert highlighted `<span>`; the others are links. A non-default `sort` also lights the funnel gold (it's part of `filtersActive()`).
- **Filters / state**: the album (path) + `min_rating` + media toggles + `aspect` + `recursive` + `tags` + `sort` (query) are the SPA's state, read from the URL on load and written back on each navigation. Every client-built breadcrumb / sub-album / star / toggle link carries the current filters so they persist while browsing; they also constrain the sub-album tile covers/counts. The server-side `Filters` struct ([src/models.rs](src/models.rs)) is used by the JSON API handlers and embedded in saved bookmarks.
- **Sub-album grid** (below the breadcrumb): direct sub-albums (newest-first, from `/api/subalbums`); each tile is the cover image with the bold sub-album name and `(count)` centered on top, linking to that sub-album. Covers use the same lazy thumbnail pipeline as the grid (below).
- **Photo grid**: grouped by day (newest first) under the `modified`/`created` sort ‚Äî by the sort's date column ‚Äî but a **flat, ungrouped grid under `sort=name`** (A‚ÄìZ); fixed-height tiles wrapping left-to-right. Image tiles are `<img class="thumb">` rendered **src-less** (the day's tiles reserve their width from the photo's aspect ratio so the layout doesn't reflow as they decode in); an `IntersectionObserver` then loads each `/api/photos/:id/thumbnail` (raw PGF), decodes it in a webpgf **wasm Web Worker**, and paints it ‚Äî rotated per `X-Orientation` ‚Äî via a canvas‚Üíblob URL. A `404` or any decode failure falls back to the full-size `/file` (`data-full`); that URL is also what the lightbox opens. Videos (`is_video`) are a `<button class="vtile">` with a ‚ñ∂ badge and an **inner poster** `<img class="thumb">` fed by the same pipeline (no `data-full`, so a missing thumbnail just leaves the ‚ñ∂ placeholder; the video itself isn't fetched until opened). The lightbox enumerates only *direct* grid children (`.grid > img, .grid > .vtile`), so a poster isn't its own item. See [Thumbnails](#thumbnails).
- **Keyboard navigation** (`initGridNav` in [gridnav.js](src/web/gridnav.js)): the **arrow keys** move a highlighted selection (`.selected`, an inset gold outline) across **both** grids in one sequence ‚Äî `.albums > a.album` then `.grid > img.thumb, .grid > .vtile`. Left/Right step in DOM (reading) order; Up/Down are *geometric* (nearest row above/below by tile center, then closest column ‚Äî the grids wrap with a variable column count); **Home/End** jump to the first/last tile, and **PageUp/PageDown** move by about a viewport (row-by-row in the same column). **Enter** activates the selection via `selected.click()` (so it reuses the existing delegated handlers: open the lightbox for a photo/video, SPA-navigate into a sub-album). The selection is mirrored to the **URL fragment** ‚Äî `#item-<photoid>` for photos, `#item-<encoded name>` for sub-albums (each tile carries that `id`) ‚Äî via `replaceState` (no history spam), and `restoreSelection()` re-applies it after every render, so it survives reload / Back/Forward. Guards: inert while the lightbox is open or a modifier is held (so **Alt+‚Üë** parent-nav still works). The **lightbox tracks the selection too**: every `show()` (arrows / Home/End / wheel / swipe / random / slideshow) calls `setSelected(tiles[i])`, and `dismiss()` writes that last-viewed item to the grid selection + fragment ‚Äî so closing the lightbox lands on the item you were viewing.

### Lightbox

Click a photo/video to open it full-page over a dimmed grid, requesting **fullscreen** (Fullscreen API; guarded/no-op where unsupported, e.g. iPhone Safari). The media is scaled to fill the viewport (up or down, preserving aspect). **Fullscreen is decoupled from open/closed**: the **`f` key** toggles fullscreen (`toggleFullscreen`), and losing fullscreen to a **tab switch** or the `f` toggle leaves the lightbox open as a full-viewport overlay. Esc still dismisses even in fullscreen ‚Äî the browser swallows the Esc *keydown* to exit fullscreen, so a `fullscreenchange` handler does the dismiss, distinguishing a deliberate exit **while the tab keeps focus + visibility** (dismiss) from a tab switch (backgrounded ‚Üí focus/visibility lost) or our own `f` toggle (a flag), both of which keep it open.

- **Videos**: auto-play, looping, with **native `controls`** (play/pause/seek/volume; mp4/webm only); Space toggles play/pause, `m` toggles mute. On **touch**, swipes take over navigation, but a **tap on a video is left to the browser** ‚Äî it shows/hides the native control overlay (play/pause/seek live there, not in our tap handling).
- **Navigation**: keyboard ‚Üê/‚Üí (`preventDefault` stops a focused video from also seeking), Home/End ‚Üí first/last, `r` ‚Üí a random item, the on-screen ‚Äπ ‚Ä∫ chevrons, the **mouse wheel** (down = next; throttled to one item per notch), and **touch** swipe left/right (anywhere, incl. over a video); swipe up ‚Üí random. Opened by keyboard (`t`/`m`) or by the top-left action row's Tag/Move buttons (mouse/touch).

On apply failure (e.g. `403`) the picker stays open with pending state intact ‚Äî Enter retries, Esc bails. MRU checkbox rows are **mirrored** to their tree twin by tag id (a delegated `change` listener, checkboxes only); the apply diff reads only the tree.

Activating the current album just closes. **Enter in the filter** moves to the **first real match** (never a dim `.ctx` ancestor) and only with a **non-empty query** ‚Äî a stray Enter on the empty filter can't move anything. On success the tile's `album_path` is updated in place and `metaCache[id]` is **dropped** (its `file_path` changed; the open panel refetches lazily) ‚Äî the grid tile deliberately stays until the next re-render (removing it live would desync the open lightbox's items). Keyboard: ArrowDown from the filter focuses the first visible row, ArrowUp/Down walk MRU + tree rows as one sequence (ArrowUp from the first returns to the filter), focus is native (`:focus-within` gold outline). Guards: while open, the picker owns the keyboard (lightbox bindings are inert), the wheel scrolls the list instead of flipping photos, touches on it are left native (`touch-action: pan-y` re-enables scrolling), and a fullscreen Esc (which the browser swallows ‚Äî it lands in the `fullscreenchange` handler) closes the picker, not the lightbox (fullscreen is lost ‚Äî a browser security behavior pages can't prevent cross-browser; `f` re-enters).
- **Slideshow** (`s` or the ‚ñ∂/‚è∏ button in the top-left action row ‚Äî kept out of the bottom edge so it stays clear of a video's native controls): auto-advances to a **random** item ‚Äî an image after **5s**, a video after it **plays in full** (its `loop` is turned off so `ended` fires; an unplayable one advances after 1.5s). Dismissing the lightbox stops it.
- **Dismiss**: click the letterbox / Esc / the X / the device Back button, or a deliberate fullscreen exit **while the tab is visible** (Esc or the browser's own control) ‚Äî but *not* the `f` toggle or a tab switch (see above). Opening pushes a history entry **carrying no URL** (so the album URL is preserved); the single shared `popstate` handler dismisses the lightbox when open (leaving the album in place) and otherwise treats the pop as album Back/Forward (re-render). On dismiss, the grid scrolls the last-viewed tile into view if browsing left it off-screen (`scrollIntoView({block:'nearest'})` with `scroll-margin-top` = the sticky navbar height, so it isn't tucked underneath).
- **Perf**: preloads the prev/next images, `decoding="async"`, and `touch-action: manipulation` to cut mobile tap latency.

### Undo

A **session-only, client-side undo** ([undo.js](src/web/undo.js)): every mutating operation pushes `{label, undo}` onto an in-memory stack via `pushUndo` ‚Äî `label` is the human-readable description shown in the confirmation toast ("rating of img_1.jpg (‚òÖ‚òÖ‚òÖ‚òÜ‚òÜ ‚Üí unrated)"), `undo()` reverses the operation and returns a Promise (reversal is itself a server call). The **`u` key** (anywhere in the SPA ‚Äî lightbox open or closed, since ops outlive the view that created them; inert with modifiers or while typing in an input) pops the top op and runs it: success toasts "Undone: ‚Ä¶", failure (e.g. the server now runs without `--allow-writes`) toasts red and **puts the op back** so `u` retries it. The manager is generic ‚Äî it knows nothing about ratings; future operations (tag edits, moves, ‚Ä¶) just push their own ops. Deliberately no server-side or persisted state (reload forgets the stack), and no redo. The **toast** (`#toast`, fixed bottom-center, above the lightbox) lives here too and doubles as the rating-change confirmation; since a fullscreen element (the lightbox) is the only thing the browser paints, `toast()` (re)hosts the element inside `document.fullscreenElement` when set, else `document.body`.

### Installable PWA
The app is a Progressive Web App, so it can be "installed" (Android home screen, desktop, etc.). Embedded + served like the other static assets: `GET /manifest.webmanifest` (the manifest: `digiKam Browse`, `display: standalone`, `start_url: /photos`, theme/background `#1a1a1a`/`#111`), `GET /icon-192.png` & `GET /icon-512.png` (rasterized from digiKam's `digikam_oxygen.svg`), and `GET /sw.js` (the service worker). The `<head>` carries `<link rel="manifest">`, `<meta name="theme-color">`, and an `apple-touch-icon`; [main.js](src/web/main.js) registers the service worker on `load`. The SW is **deliberately a no-op for fetches** (empty `fetch` handler): a response served from a service worker **bypasses the browser's HTTP cache**, which would defeat our `Cache-Control` headers (immutable assets, the `/photos` `max-age`), so every request is left to the browser. It exists only to satisfy the PWA installability requirement (and as the place to add real offline support later, if its caching is made to respect those headers). The manifest and `sw.js` are served `no-cache` (revalidated via content-hash `ETag`) so updates propagate; icons are `immutable`.

> **Installation needs a secure context.** Service workers only register over **HTTPS** (or `localhost`). Reaching the server from a phone over plain `http://<lan-ip>:8080` is **not** a secure context, so the SW won't register and the browser won't offer "Install". Put it behind TLS (an nginx reverse proxy / `tailscale serve` / a tunnel) for a trusted cert and a secure context.

## Architecture & design choices

- **Stack**: `axum` + `tokio`; `rusqlite` (feature `bundled`, so SQLite is compiled in ‚Äî no system lib / pkg-config) behind an `r2d2` connection pool. rusqlite is blocking, so every DB call runs inside `tokio::task::spawn_blocking` (see `run_blocking` in [src/handlers.rs](src/handlers.rs)). The app is otherwise network-isolated except for **one outbound call**: the Yandex reverse-search upload (`reqwest` with `rustls-tls` + bundled webpki roots, so no system TLS/OpenSSL dep), used only by `/api/photos/:id/reverse-search`.
- **HTML rendering**: the page **shell** uses [`maud`](https://maud.lang.rs/) (compile-time `html!` templates with automatic escaping). `album_page` returns `maud::Markup` (its axum feature makes it `IntoResponse`). The shell is static ‚Äî all album/filter rendering moved client-side ([src/web/](src/web/)), so maud now just emits the fixed navbar/container scaffolding. `assets/style.css` and the `concat!`-ed `web/*.js` (`STYLE` / `SCRIPT`) are emitted inside `<style>`/`<script>` via `PreEscaped` (trusted, must not be escaped).
- **Read-only by default & safe alongside running Digikam**: connections open with `SQLITE_OPEN_READ_ONLY`, set `PRAGMA query_only=ON`, and a 5s `busy_timeout` so reads don't fail while Digikam writes. We deliberately do **not** use `immutable=1` (Digikam may be modifying the file concurrently). With the opt-in `--allow-writes` the same single pool opens `SQLITE_OPEN_READ_WRITE` (never CREATE) without `query_only`, `AppState.allow_writes` gates the write handlers (`403` otherwise), and the `busy_timeout` makes our writes contend politely with Digikam's. See `build_pool` in [src/db.rs](src/db.rs).
- **One writable DB ‚Äî `web.sql`** (bookmarks): the sole exception to read-only. `build_web_pool` in [src/db.rs](src/db.rs) opens *our own* `web.sql` with `SQLITE_OPEN_READ_WRITE | SQLITE_OPEN_CREATE` and **no** `query_only`, then runs a `CREATE TABLE IF NOT EXISTS bookmarks(...)` migration. It's optional (`AppState.web: Option<Pool>`, graceful like `thumbs`); the bookmark handlers run via a `run_web` helper. Never points at a Digikam DB.
- **Path resolution** ([src/db.rs](src/db.rs)): an image's absolute path is `AlbumRoots` base + `Albums.relativePath` + `/` + `Images.name`. The root base is parsed from the album-root identifier (percent-decoded), joined with `specificPath`: local volumes use `volumeid:?path=‚Ä¶&fileuuid=‚Ä¶`, network shares use `networkshareid:?mountpath=‚Ä¶&fileuuid=‚Ä¶` ‚Äî we accept either `path=` or `mountpath=` (a root whose identifier has neither is skipped with a warning). The root album has `relativePath == "/"`.
- **Visibility filter**: only `Images.status = 1` (visible) is returned; 3 (trashed) and 4 (obsolete) are excluded.
- **Ratings/dimensions**: `rating`, `width`, `height`, `file_size`, `id` are `u64`. Digikam stores `-1` for "unrated"; that (and any negative) maps to JSON `null`.
- **Tags tree** ([src/handlers.rs](src/handlers.rs)): built in memory from `Tags(id, pid, name)`; top-level nodes have `pid = 0`. The internal `_Digikam_Internal_Tags_` subtree (id `1`, which holds Color/Pick-label tags) is excluded from `/tags`.
- **Request logging**: a `tower_http` `TraceLayer` logs every HTTP request (method + URI) and its response (status + latency) at `info`. `--trace-sql` additionally logs the SQL each request runs.
- **No CORS layer**: the frontend is served same-origin by this server, so CORS is unneeded. It was removed because `CorsLayer::permissive()` emits `Vary: origin, ‚Ä¶`, which made the browser refuse to reuse the cached (immutable) thumbnails and webpgf assets ‚Äî it re-`GET`s them every page load. (Re-add a scoped CORS layer only if a *separate-origin* frontend ever consumes `/api`.)

### Relevant Digikam schema
- `AlbumRoots(id, label, identifier, specificPath)` ‚Äî collection roots.
- `Albums(id, albumRoot, relativePath)` ‚Äî directories; unique `(albumRoot, relativePath)`.
- `Images(id, album, name, status, fileSize, uniqueHash, modificationDate)` ‚Äî we order and day-group by `modificationDate`.
- `ImageInformation(imageid, rating, creationDate, width, height, format, ‚Ä¶)` ‚Äî `creationDate` is the import time; unused for ordering (see above).
- `Tags(id, pid, name)`; `TagsTree(id, pid)` is the ancestor transitive closure (`SELECT id FROM TagsTree WHERE pid = T` gives descendants of `T` ‚Äî currently unused because tag matching is exact).
- `ImageTags(imageid, tagid)`, `ImagePositions(imageid, latitudeNumber, longitudeNumber)`.

### Thumbnails
Digikam's `thumbnails-digikam.db` blobs are **PGF** (a wavelet format browsers can't decode natively) and its `FilePaths` are stale (`/mm/Images/‚Ä¶` vs the real `/home/eelco/Images/‚Ä¶`) ‚Äî so we key on `uniqueHash`+`fileSize`, not paths. Approach: `/api/photos/:id/thumbnail` streams the raw PGF blob untouched; the **client decodes it in wasm** via [`webpgf`](https://github.com/haplo/webpgf) (built by [nix/webpgf.nix](nix/webpgf.nix), `nix build .#webpgf`). webpgf already maps libpgf's BGRA ‚Üí RGBA and yields an `ImageData`; it does **not** apply orientation, so the client must rotate per the `X-Orientation` header. Thumbnails are ‚â§256 px, ~19 KB avg, with near-full coverage.

The webpgf module is **embedded into the binary** (`include_bytes!`, like the CSS/JS) and served at `GET /webpgf.js` (`text/javascript`) and `GET /webpgf.wasm` (`application/wasm`, so the browser can stream-compile). Both carry a content-addressed `ETag` (the webpgf nix store hash) honoring `If-None-Match`‚Üí`304`, plus a 1-year `immutable` `Cache-Control` (`public, max-age=31536000, immutable`, shared with the thumbnails ‚Äî `IMMUTABLE_CACHE_CONTROL`) so the browser stops re-requesting them every page load. The embed path comes from the `WEBPGF_PATH` env var, which the flake sets to the `webpgf` derivation output for **both** `nix build` (`commonArgs`) and the dev shell ‚Äî so plain `cargo build` inside `nix develop` embeds them too.

The frontend ([thumbnails.js](src/web/thumbnails.js)) wires this up. An `IntersectionObserver` with a wide, viewport-relative `rootMargin` (‚âà1 screen above, ‚âà2.5 below) triggers a `/thumbnail` fetch per `img.thumb` tile **well before** it scrolls in, so paging down lands on already-decoded images. Fetch (network) and decode (CPU) are **separate stages**: each finished PGF blob queues for the next idle worker in a small **Blob Web Worker pool** (`min(hardwareConcurrency, 6)`).

First-paint latency was dominated by **Firefox network behavior** (full write-up in [docs/thumbnail-loading-performance.md](docs/thumbnail-loading-performance.md)), fixed three ways (see [thumbnails.js](src/web/thumbnails.js) ‚Äî verified to take first paint from ~500ms to ~120ms):

1. **Cap concurrent fetches** at 6 ‚Äî firing a screenful at once makes Firefox's request pacer hold the *whole burst* for hundreds of ms.
2. **`priority: 'high'`** on every fetch ‚Äî Firefox otherwise deprioritises `fetch()` for ~150ms during page load.
3. **Kick off the first ~24 tiles synchronously** at script start rather than from the observer callback ‚Äî requests issued during initial parse are dispatched promptly, later ones are held (the observer handles the rest, skipping the eager ones).

To make the *first* paint snappy, the decoder assets are fetched **once** (not once per worker): `/webpgf.js` is inlined into the worker body and `/webpgf.wasm` is fetched once, then the pool is **pre-warmed** ‚Äî each worker instantiates the module at page load (via an init message carrying the shared wasm bytes), overlapping with the thumbnail fetches, so blobs decode the instant they arrive rather than waiting on a cold module. Each worker returns an `ImageData` whose buffer is transferred back. The main thread draws each result ‚Äî applying the EXIF `X-Orientation` (2..8; 0/1/junk = no rotation) ‚Äî to a canvas, then sets the `<img>` to the canvas's blob URL. The worker URLs are made **absolute** (`location.origin` baked in) because a `blob:` worker resolves relative paths against its opaque blob base, not the page origin. A worker that dies (e.g. webpgf failed to load) falls back its tile to `/file`; once the whole pool is gone, queued/new tiles do too.
